# Vort — Bitcoin Fixed-Income Infrastructure

Welcome to the Vort hackathon repository! This guide is designed to help new teammates understand the architecture, run the project locally, and get up to speed with both the smart contracts and the frontend components.

Vort is a protocol on the Stacks blockchain that transforms yield-bearing assets (like sBTC) into tradable fixed-income primitives. It works by splitting the deposited asset into a **Principal Token (PT)** and a **Yield Token (YT)**.

---

## 🏗️ 1. Architecture Overview

At a high level, Vort consists of two main parts:

1. **Smart Contracts (Clarity on Stacks)**: Handles the core logic — depositing, minting tokens, redeeming, distributing yield, and the AMM for trading.
2. **Frontend (Next.js)**: A React application that provides the user interface for depositing sBTC, trading PTs, and managing portfolios. It uses `@stacks/connect` and `@stacks/transactions` to interact with Leather/Xverse wallets.

### 📜 Smart Contracts

All contracts are located in the `/contracts` directory.

- **`tokenizer.clar`**: The main entry point. Users call `deposit` here with sBTC. It locks the sBTC and mints 1 PT and 1 YT for the user. After maturity, users call `redeem` to burn their PT/YT and get sBTC back.
- **`pt-token.clar` (Principal Token)**: An SIP-010 standard fungible token. Represents the claim to 1 sBTC at maturity.
- **`yt-token.clar` (Yield Token)**: An SIP-010 token. Holders of YT are entitled to claim the protocol yield generated by the locked sBTC.
- **`pt-amm.clar` (Automated Market Maker)**: The decentralized exchange for trading sBTC and PT. Contains functions like `swap-sbtc-for-pt` and `swap-pt-for-sbtc`. This allows users to lock in fixed yield early by selling PT.
- **`yield-router.clar`**: Manages the accumulation of yield (from external protocols like Zest) and handles the distribution to YT holders when they call `claim-yield`.
- **`sbtc-token.clar`**: A mock/test implementation of sBTC used for testnet and sandbox environments. Includes a `mint-for-testing` function so we can get fake sBTC.

### 🖥️ Frontend (Next.js)

The frontend is built with React, Next.js (App Router), and Tailwind CSS.

- **`/src/app/dashboard`**: Contains all the core application pages (Overview, Deposit, Portfolio, Market, Protocol Health).
- **`/src/components`**: Reusable UI components (Navbar, buttons, generic layout wrappers).
- **`/src/lib/stacks.ts`**: **CRITICAL FILE.** This contains all the client-side functions that actually trigger Leather wallet popups to call our smart contracts (e.g., `depositSbtc()`, `swapSbtcForPt()`).
- **`/src/lib/hiro.ts`**: The backend API layer that talks to the Hiro API. It fetches real-time Stacks blockchain data (like balances).
- **`/src/hooks/useBalances.ts`**: A custom React hook that wraps the Hiro API. The dashboard pages use this to get the user's real sBTC, PT, and YT balances.

## 🚀 2. Local Development Setup

Follow these steps to get everything running on your own machine.

### Prerequisites

1. **Install Node.js** (v18 or higher recommended).
2. **Install a Stacks Wallet**: You need [Leather Wallet](https://leather.io/) or [Xverse](https://www.xverse.app/) installed as a browser extension.
3. Make sure your wallet is set to **Testnet** mode in its settings.

### Installation

1. Clone this repository (or pull the latest `main` branch).
2. Open your terminal in the `vort` folder.
3. Install the dependencies:
   ```bash
   npm install
   ```

### Running the Frontend

Start the Next.js development server:

```bash
npm run dev
```

The app will be available at `http://localhost:3000`.

- `http://localhost:3000` is the marketing landing page.
- `http://localhost:3000/dashboard` is the actual application.

## 🧪 3. How to Test the Application

Since we are building on the Stacks Testnet, here is the full flow you should walk through to verify everything works:

### Step 1: Connect Wallet

- Go to `http://localhost:3000`.
- Click **"Connect Wallet"**. Your Leather/Xverse wallet should pop up requesting connection.
- Once connected, you will be redirected to the Dashboard.

### Step 2: Get Test sBTC

Your dashboard will show 0 balances initially. Before you can deposit, you need test sBTC.

- Go to the **Deposit** page (`/dashboard/deposit`).
- Because your balance is 0, a banner will appear offering to "Mint 1 Test sBTC".
- Click the button. Your wallet will pop up asking you to sign the `mint-for-testing` transaction on the `sbtc-token` contract.
- Approve it and **wait about 30 seconds to 1 minute** for the transaction to confirm on the testnet.
- _The frontend automatically polls the Hiro API every 5 seconds after you sign a transaction, so your balance will update automatically once confirmed._

### Step 3: Deposit sBTC

- Once you have sBTC, stay on the Deposit page.
- Enter an amount (e.g., 0.5) in the input field.
- Click **"Sign Transaction"**.
- Your wallet will pop up to sign the `deposit` function on the `tokenizer` contract.
- Approve it and wait for confirmation. Once confirmed, you will receive PT and YT tokens.

### Step 4: Verify Portfolio & Market

- Check the **Overview** or **Portfolio** page. You should see your new PT and YT balances.
- Go to the **Market** page. You can test the AMM by swapping sBTC for PT, or PT for sBTC. (Note: The AMM requires liquidity to function properly, so swaps might fail or have high slippage if the pool is empty).

## 🧑‍💻 4. Where to Make Changes

If you need to edit specific parts of the app:

**To edit smart contracts:**

- Open the `.clar` files in the `/contracts` directory.
- If you change a contract, you will need to re-deploy it to the testnet using Clarinet and update the `NEXT_PUBLIC_CONTRACT_DEPLOYER` variable in the frontend if the address changes.

**To edit contract calls (Wallet Popups):**

- Open `/src/lib/stacks.ts`. If you add a new contract function, add a new helper function here following the same pattern.

**To edit the Dashboard UI:**

- The Overview page is `/src/app/dashboard/page.tsx`.
- The Deposit interface is `/src/app/dashboard/deposit/page.tsx`.

**To modify how balances load:**

- Backend API logic: `/src/app/api/balances/[address]/route.ts` and `/src/lib/hiro.ts`.
- Frontend state: `/src/hooks/useBalances.ts`.

## ❓ FAQ & Troubleshooting

**"My balance isn't updating after depositing."**
Stacks is a Bitcoin layer. Transactions on the testnet are much faster than mainnet, but they still take 15-45 seconds to confirm in a block. The frontend polls every 5 seconds to check. If it takes longer, you can check your address on the [Hiro Explorer (Testnet)](https://explorer.hiro.so/?chain=testnet) to see if the transaction is still pending.

**"The AMM Swap says 'Transaction Failed' in my wallet."**
If the AMM pool hasn't been initialized with liquidity by an admin (or if someone drained the test liquidity), swaps will error out. This is expected behavior for an empty AMM.

**"What is the 5.42% APY based on?"**
Currently, because the testnet AMM lacks constant liquidity, the APY in the UI is hardcoded as an estimate (`~5.42%`). In a real production environment, this number would be dynamically calculated by reading the current `PT` price from the AMM pool.
